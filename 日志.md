5. 对你的项目来说，更合适的原则是：一个任务对应一个清晰的责任边界，且这个边界最好能讲成“采样 / 解析 / 融合 / 控制 / 执行 / 保护”中的一个环节。
6. 如果两个环节之间只是简单传值，而且没有独立周期、独立失效处理、独立阻塞等待，那它们就可以合并，没必要为了“看起来像大工程”故意切开。

8. 你现在最需要的不是盯着别人的任务个数做心理参照，而是把自己的主链路定清楚：输入、融合、控制、执行、保护。先把这五段切明白，任务数量自然会比较合理。

2026-03-28 关于 FreeRTOS stream buffer 的 API 角色

1. `stream buffer` 可以先记成两端：生产者用 `xStreamBufferSend` 或 `xStreamBufferSendFromISR` 投递字节，消费者用 `xStreamBufferReceive` 阻塞取字节。
2. 对这个项目最典型的场景是：UART 接收中断或 DMA 回调把新到的数据塞进 stream buffer，`task_remote_control` 里阻塞等待并批量取出后再喂给 `sbus`。
3. 这类接口比手写轮询更适合 RTOS，因为它天然表达了“谁在等数据、谁在投数据”。
4. 你可以把它理解成“任务边界的字节管道”，而不是“通用内存容器”。
5. 如果后面要我给代码例子，优先会演示 `CreateStatic` + `SendFromISR` + `Receive` 这条最贴近你项目的用法。
2026-03-28 关于 task_remote_control 使用 StreamBuffer 的归属

1. `task_remote_control` 是这个 `stream buffer` 的 owner，最合适的做法是由它在初始化阶段创建并持有句柄。
2. UART 中断回调只负责把新收到的字节投进去，应该用 `xStreamBufferSendFromISR`，不要把 `stream buffer` 的创建职责放到 `bsp_uart` 里。
3. `task_remote_control` 任务本身再用 `xStreamBufferReceive` 阻塞取出数据，然后喂给 `sbus` 解析器。
4. `stream buffer` 的静态存储区和控制块可以先放在 `task_remote_control` 这个 App 文件里，后面如果要做成多输入源，再考虑把它提升到更通用的 `input` 上下文。
5. 这条链路的职责最清楚的形态是：`bsp_uart` 产出事件，`task_remote_control` 拥有 buffer 和任务，`sbus` 只吃字节流。
2026-03-28 关于 task_remote_control.c 里的 static stream buffer 存储

1. 文件里的 `static` 仍然有意义，因为它表达的是“这个 buffer 只属于当前编译单元”，不是“给别处 extern 用”。
2. 如果去掉 `static`，这些符号就变成外部可见，虽然你不写 `extern` 也能编译，但封装边界会变弱，也更容易被别的文件误用。
3. 对当前这个 `stream buffer` 来说，最重要的不是能不能外部访问，而是它到底归谁所有；既然归 `task_remote_control`，留在 `task_remote_control.c` 里做文件私有静态对象是合理的。
4. 但更推荐的写法是把 `StreamBufferHandle_t` 也放进任务上下文里，而不是只让静态存储区私有、句柄却留成局部变量。
5. 所以结论是：`static` 不只是为了防 `extern`，而是为了表达 owner 关系和隐藏实现细节；在这个场景下可以保留。
2026-03-28 关于 StreamBuffer 是否只给 remote_control 用

1. 是的，按现在这个结构，这个 `StreamBuffer` 先只给 `remote_control` 这条输入链路用。
2. 它的职责是把 UART 事件和 `task_remote_control` 解耦，不是给其他任务做通用消息总线。
3. 其他任务如果要拿遥控结果，不应该直接读这个 `StreamBuffer`，而应该读 `task_remote_control` 解出的统一控制快照，或者等后面再做更高层的消息发布机制。
4. 如果以后出现多个控制来源，那么通常是新增别的输入通道或别的 buffer，而不是让一个 `StreamBuffer` 同时服务所有任务。
5. 现在先把它理解成“遥控输入专用管道”最合适。
2026-03-28 关于 xStreamBufferSendFromISR 的参数

1. `xStreamBufferSendFromISR` 的第四个参数需要传 `BaseType_t *pxHigherPriorityTaskWoken`，不能直接传 `(BaseType_t)pdTRUE`。
2. 这个值通常先定义成局部 `BaseType_t xHigherPriorityTaskWoken = pdFALSE;`，然后把它的地址传进去，必要时再在 ISR 末尾做切换。
3. `rx_event` 这个参数如果当前没用到，至少要显式忽略，避免告警。
4. 其余写法整体思路没问题，核心是“中断回调里把字节段投给 StreamBuffer”。
2026-03-28 关于 xHigherPriorityTaskWoken 和 portYIELD_FROM_ISR

1. `xHigherPriorityTaskWoken` 的作用是告诉 ISR 末尾：这次从 ISR 里唤醒了没有更高优先级任务。
2. `xStreamBufferSendFromISR` 在成功唤醒等待接收的高优先级任务时，会把这个变量置为 `pdTRUE`。
3. `portYIELD_FROM_ISR(xHigherPriorityTaskWoken)` 的作用是让系统在中断返回后尽快切到那个更高优先级任务，而不是等到下一个 tick。
4. 如果没有任何更高优先级任务被唤醒，这个变量保持 `pdFALSE`，那这一句就不会引发额外调度。
5. 同一个 ISR 里多次调用 `FromISR` API 时，通常复用同一个 `xHigherPriorityTaskWoken`，最后只切一次。
2026-03-28 关于 StreamBuffer 是否能给多个任务共享

1. `StreamBuffer` 适合单写单读，不适合拿来当多个任务都能直接访问的共享消息通道。
2. 对当前项目，最自然的用法仍然是“一个输入源对应一个 `task_remote_control` 的接收链路”。
3. 其他任务如果需要遥控结果，应读 `task_remote_control` 产出的统一控制快照，而不是直接读这个 `StreamBuffer`。
4. 如果以后真有多个输入源，通常是新增别的 `StreamBuffer` 或更高层的消息机制，而不是让一个 buffer 同时服务所有人。
5. 所以现在它先只服务遥控输入链路，这样边界最清楚。
2026-03-28 关于 task_remote_control 当前结构是否足够

1. 现在这版思路是对的：`task_remote_control` 持有 `StreamBuffer`、`sbus` 解析器和 `rc_mapper`，职责边界已经比较清楚。
2. 但它还缺一件关键事：初始化后要真正启动 UART 接收，否则回调不会进来，`StreamBuffer` 里也不会有数据。
3. 另外，`taskRemoteControlUpdate()` 现在是阻塞消费接口，更适合放在任务主循环里，而不是到处被外部随便调用。
4. `taskRemoteControlGetRCMapped()` 更适合返回只读视图，避免外部任务直接修改内部状态。
5. 这版结构可以继续往下写，但必须补上“谁调用 init、谁启动接收、谁在任务里循环 update”这三个点，才能真正跑起来。
2026-03-28 关于返回只读指针

1. 如果想保护内部状态，函数返回类型应该写成 `const moduleRCMapper_t *`，也就是“指向常量的指针”。
2. 这能阻止外部通过返回值直接修改 `task_instance_` 里的 `rc_mapper_`。
3. 对应地，函数参数里如果只是读取，也应该尽量用 `const` 限定输入。
4. 如果外部确实需要修改，应该提供专门的更新接口，而不是把内部地址直接暴露成可写指针。
2026-03-28 关于 task_remote_control 的 StreamBuffer 回调修正

1. 回调里要先检查 `owner_ptr`、`stream_buffer_handle_` 和 `rx_buffer_ptr`，避免初始化顺序或异常状态直接崩掉。
2. `xStreamBufferSendFromISR` 要检查返回值，缓冲满时可能只写入一部分字节，最好顺手累计丢包计数。
3. `xStreamBufferCreateStatic` 的静态存储区长度要按 `xBufferSizeBytes + 1` 准备，不能只按逻辑容量分配。
4. `taskRemoteControlUpdate` 这类接口最好显式带 `TickType_t` 超时参数，和 FreeRTOS 的阻塞语义保持一致。
5. 这版写法已经比“把回调直接裸写进 StreamBuffer”更稳，也更适合作为 App 层输入管道。
2026-03-28 关于 remoteControlTask 的最小启动测试

1. 最小测试方式是在 `MX_FREERTOS_Init()` 里直接创建 `remoteControlTask`，然后在任务函数里先调用 `taskRemoteControlInit()`。
2. 任务循环里用 `taskRemoteControlUpdate(pdMS_TO_TICKS(10))` 阻塞收数，再配一个很短的 `osDelay(1)`，这样既能验证任务启动，也能验证输入链路。
3. 这个测试的重点不是遥控数据是否已经完整解析，而是确认 `App -> task -> StreamBuffer -> SBUS` 这一整条链路能跑起来。
4. 如果后面要更容易观测，可以再加一个调试计数器或 LED 心跳，但第一步先确保任务能被调度到。
2026-03-28 关于 remoteControlTask 是否还需要额外延时

1. 如果 `taskRemoteControlUpdate()` 已经使用 `portMAX_DELAY` 或足够长的阻塞时间等待 `StreamBuffer`，那外层循环一般不需要再加 `osDelay(1)`。
2. 这种写法会变成事件驱动：没数据时任务阻塞，有数据时被唤醒处理，然后再次阻塞。
3. 额外的短延时通常只会增加响应延迟，不会提升结构质量。
4. 只有当这个任务还要顺带做周期性检查、心跳或别的低频工作时，才需要再给它单独的节拍。
5. 对遥控输入链路来说，优先让它“来数据就处理，没数据就睡”，更贴合 RTOS 思路。
2026-03-28 关于 remoteControlTask 最小测试循环的写法

1. 只要目的是验证任务能跑、UART 回调能进、StreamBuffer 能收，`for(;;){ taskRemoteControlUpdate(portMAX_DELAY); }` 这种循环就够了。
2. 如果只是把摇杆值拷贝到局部变量里但后面不用，这些局部变量可以先删掉，避免把测试代码写得像正式逻辑。
3. 如果是为了断点观察数据，建议直接看 `taskRemoteControlGetRCMapped()` 返回的结构体，而不是再复制一份临时变量。
4. 真正要做失控保护时，就不要无限阻塞，要改回有限超时。
2026-03-28 关于 stream buffer 持续 drop 的真实原因

1. 这次最关键的原因不是回调逻辑本身，而是 `xStreamBufferCreateStatic` 只给了很小的有效容量，远小于 UART DMA 可能一次送进来的数据量。
2. `taskRemoteControlUpdate` 之前每次只取 25 字节，也会让积压更容易发生。
3. SBUS 数据虽然被丢了一部分，但因为协议有固定帧头和重同步能力，所以摇杆值仍然可能看起来是对的。
4. 解决思路是同时增大 stream buffer 容量，并让任务每次尽量把已经积压的数据读干净。
5. 这类问题本质上是“生产者批量输入速度”和“消费者每次处理量”不匹配，而不是单纯的语法或回调错误。
2026-03-28 关于稳定丢 4 字节的根因

1. 根因不是单纯的 buffer 太小，而是生产者一次写入的字节数和消费者一次读取的字节数长期不匹配。
2. 你现在的链路是“UART 回调把一段字节送进 StreamBuffer，任务每次只取 25 字节就返回”，这会让剩余字节不断在 StreamBuffer 里累积。
3. `StreamBuffer` 是字节管道，不是消息队列；如果不把当前可读数据一次性尽量读干净，残留会跨轮次叠加。
4. 当累计残留接近缓冲区上限时，新的 UART 回调就会出现稳定的部分写入，表现为每次固定丢几个字节，这次看到的就是 4 字节。
5. 真正的修正方向是让任务侧在一次唤醒后尽量把 buffer 读空，而不是每次只取固定 25 字节就立刻返回睡眠。
2026-03-28 关于 8E2、IDLE 中断和稳定丢 4 字节的关系

1. 8E2 说的是“每个 UART 字节”的格式，不是“协议帧结束就自动触发空闲中断”。
2. IDLE 中断的条件是 RX 线在最后一个 stop bit 之后继续保持空闲一段时间，它和 stop bit 数量不是一回事。
3. SBUS 的帧边界通常会带来空闲间隔，所以常常能在帧间触发 IDLE，但这不保证每次回调都刚好是 25 字节。
4. 当前稳定丢 4 字节，更像是“UART 回调送进来的字节段长度”和“任务每次消费的字节段长度”不一致，导致残留逐轮累积。
5. 解决问题要看整条链路，而不是只看 8E2：UART DMA 回调、StreamBuffer 写入、任务读取、SBUS 解析，任何一层只取固定 25 字节都会让残留积压。
2026-03-28 关于稳定丢 4 字节的直接根因

1. 现在这次的直接根因是 `xStreamBufferCreateStatic(PROTOCOL_SBUS_FRAME_SIZE * 5U, ...)` 只给了 125 字节的逻辑容量，而 FreeRTOS stream buffer 的可用空间实际只有 `xBufferSizeBytes - 1`，也就是 124 字节。
2. 上游 UART DMA 一次回调给出的是 128 字节，所以 `xStreamBufferSendFromISR` 最多只能写入 124 字节，剩下 4 字节就稳定丢掉了。
3. 这和 8E2、IDLE 中断本身没有直接关系，问题发生在“UART 回调 -> StreamBuffer”这一步的容量限制上。
4. 如果想让一次 128 字节的输入不丢，StreamBuffer 的逻辑容量至少要大于 128，也就是创建参数要比 128 再多 1。
5. 这类问题最好先看 producer 一次最多送多少，再决定 buffer 的逻辑容量，而不是按协议帧长去估算。
2026-03-28 关于 StreamBuffer 长度参数的更正

1. `stream_buffer_[PROTOCOL_SBUS_FRAME_SIZE * 5U + 1U]` 里的 `+1U` 不是多余的，它是在满足 FreeRTOS stream buffer 需要的“存储区比逻辑容量多 1 字节”要求。
2. `xStreamBufferCreateStatic(PROTOCOL_SBUS_FRAME_SIZE * 5U, ...)` 里传进去的 `PROTOCOL_SBUS_FRAME_SIZE * 5U` 是逻辑容量，不是静态数组长度。
3. 所以这里真正的关系是：逻辑容量 125 字节，静态数组 126 字节，实际可用空间 124 字节。
4. 这也解释了为什么一次送 128 字节时，`bytes_sent` 会稳定变成 124。
5. 以后判断这类问题，要分清“数组长度”“CreateStatic 传入的容量”“实际可用容量”三个概念。
2026-03-28 关于关掉遥控发射机后仍然触发回调

1. 这不一定是“循环 DMA 出错”，更常见的情况是 SBUS 接收机在发射机关闭后仍然会继续输出帧，通常是失控保持或 failsafe 帧。
2. `HAL_UARTEx_ReceiveToIdle_DMA()` 配合 circular DMA 时，DMA 本身不会因为遥控器关了就停；只要接收线上还有字节，`RxEventCallback` 仍然会继续来。
3. 所以要先区分“发射机关闭”和“接收机是否仍在给 UART 持续吐 SBUS 帧”。
4. 真正判断链路是否还活着，应该看接收字节内容、SBUS 的 `frame_lost` / `failsafe` 标志，或者看一段时间内是否还有新帧进入，而不是只看回调有没有触发。
5. 如果接收机确实还在输出 failsafe 帧，那回调继续触发是正常现象，不代表 DMA 错了。
2026-03-28 关于 HT / TC / IDLE 连续触发

1. `HT / TC / IDLE` 连着来通常不是错误，而是 circular DMA + ReceiveToIdle 的正常分段事件。
2. `HT` 表示缓冲区半满，`TC` 表示缓冲区全满，`IDLE` 表示线路又空闲了一段时间。
3. 对于连续收到的 SBUS 字节流，这三个事件可以在同一轮接收过程中依次出现。
4. 回调里不要把它们理解成“三次重复接收同一帧”，而要理解成“同一条字节流被 DMA 以不同位置切片上报”。
5. 关键是 `rx_last_pos_` 和 `Size` 的差值是否正确，而不是事件类型本身。

2026-03-28 关于关掉发射机后仍然进 RxEvent 回调的判断

1. `HAL_UARTEx_ReceiveToIdle_DMA()` 配合 circular DMA 时，`HT / TC / IDLE` 都是正常的接收事件，只要 RX 线上还有字节变化，回调就会继续来。
2. 如果“发射机关了”但接收机还通电，很多 SBUS 接收机会继续输出失控保持或 failsafe 帧，所以回调继续触发不等于 DMA 出错。
3. 如果接收机也断电了但回调还在进，更要怀疑的是 RX 线浮空、外部反相/电平处理板噪声，或者线路上还有别的设备在驱动。
4. 真正的判定方法是看接收到的 SBUS 帧内容，尤其是 `frame_lost_flag_` 和 `failsafe_activate_flag_`，不要只看事件类型。
5. 先区分“发射机关闭”“接收机是否仍在供电”“SBUS 线是否仍有有效驱动”这三件事，才能判断是不是循环 DMA 问题。


2026-03-28 关于遥控器开关、failsafe 和中断行为

1. 现在的现象已经能说明：接收机通电时，发射机关掉后仍会继续输出 SBUS，只是 `frame_lost_flag_` 和 `failsafe_activate_flag_` 会变成 `true`。
2. 这说明链路不是 DMA 乱触发，而是接收机进入了 failsafe 输出状态。
3. 把接收机拔掉后中断不再来，说明 UART/DMA 的中断路径本身是正常的，问题不在“回调自己乱跑”。
4. 上层判断遥控可用性时，应该把 `failsafe` / `frame_lost` / 超时一起作为输入，而不是只看有没有进中断。
5. 现在最关键的是区分“有数据但处于 failsafe”和“完全没数据”，这两种状态在控制逻辑上要分开处理。
2026-03-28 关于稳定丢 4 字节的完整链路

1. 这次的 4 字节丢失不是协议问题，而是“UART 回调一次送入 128 字节”与“StreamBuffer 实际只能容纳 124 字节”之间的固定容量差。
2. 现在 `xStreamBufferCreateStatic(PROTOCOL_SBUS_FRAME_SIZE * 5U, ...)` 里的 `125` 是 `xLength`，而 FreeRTOS 的 `xStreamBufferSpacesAvailable()` 在空缓冲时只会返回 `xLength - 1`，也就是 124。
3. BSP 层 `HAL_UARTEx_RxEventCallback()` 里如果一次算出 128 字节新增数据，就会在 `xStreamBufferSendFromISR()` 这里被截断，稳定剩下 4 字节写不进去。
4. `taskRemoteControlUpdate()` 只按 25 字节一口一口读，所以只会让积压更明显，但它不是这 4 字节的直接根因。
5. 所以完整链路是：DMA 收到 128 字节 -> BSP 回调转发 -> StreamBuffer 容量只剩 124 -> 固定丢 4 字节 -> 任务端再慢读，问题持续放大。
2026-03-28 关于“HT 开启后单次最多 64 字节”的修正版判断

1. 如果 HT 中断打开，单次 RxEvent 进来的上限确实可能是 64 字节，而不是 128 字节。
2. 这并不否定丢 4 字节的现象，因为 StreamBuffer 的实际可用空间是 124 字节，两个 64 字节的连续写入合起来就是 128 字节，仍然会超出 4 字节。
3. 现在更准确的根因不是“单次写 128”，而是“任务侧没有把积压字节一次性尽量读空”，导致两个 DMA 分段在 StreamBuffer 里叠加后跨过 124 字节边界。
4. `taskRemoteControlUpdate()` 只读 25 字节，然后立刻返回，会让残留字节长期留在 buffer 里，放大这个边界问题。
5. 所以后续排查要看每次送入的 `bytes_to_send`、写入前剩余空间和任务侧实际读走了多少字节，才能确认 4 字节是在哪个时刻被挤掉的。
2026-03-28 关于当前遥控任务的内存与吞吐判断

1. 现在这版代码里，DMA 接收缓冲是 `BSP_UART_RX_BUFFER_SIZE` 的 128 字节静态数组，StreamBuffer 的 `xLength` 是 129，所以实际可用容量正好是 128 字节。
2. `taskRemoteControlUpdate()` 里的 `rx_buffer[128]` 是任务栈上的临时数组，不是长期占用的静态内存；它更值得关注的是任务栈压力，而不是和 DMA/StreamBuffer 叠加成“永久 384 字节”。
3. 如果缓冲区只比上游 burst 少 1 字节就稳定丢 1 字节，这通常不是 CPU 算力不够，而是系统已经跑在容量边界上，任何调度抖动都会把那个最小余量吃掉。
4. 真正该优化的是“消费者一次把 StreamBuffer 尽量读空”以及“给 buffer 留足余量”，而不是单纯怀疑 MCU 太慢。
5. 当前结构是合理的分层，但 `rx_buffer` 可以考虑缩小到更接近 SBUS 解析所需的最小块，避免任务栈无谓膨胀。
2026-03-28 关于 remote_control 职责边界、阻塞接口和并发风险

1. 当前 `task_remote_control.c` 虽然名字里带 `task`，但实际更像“App 层的遥控输入适配器”：它把 BSP UART、StreamBuffer、SBUS 解析和 RC 映射串起来。
2. `taskRemoteControlUpdate()` 是一个阻塞式消费接口，适合由单个任务驱动，不适合让多个上层模块随意并发调用。
3. `taskRemoteControlGetRCMapped()` 现在返回的是内部可变对象的只读指针，能减少外部误改，但并不能自动消除读取与更新之间的时序竞争。
4. `StreamBuffer` 这条链路本身是单写单读模型，当前结构下主要风险不在缓冲区本身，而在 `rc_mapper_` 被读写交错时可能出现的“撕裂读”。
5. 上层如果只需要业务指令，最好不要直接依赖原始摇杆值，而是再上一层把 SBUS/RC 状态翻译成任务可消费的语义命令。
2026-03-28 关于 App 层遥控状态机

1. 现在 `app_remote_control` 已经从“只转发摇杆值”升级成“状态 + 快照”的小状态机。
2. `APP_REMOTE_CONTROL_STATE_CONTROL` 表示收到正常 SBUS 帧，可以把 `rc_mapper_` 继续向上层输出。
3. `APP_REMOTE_CONTROL_STATE_FAILSAFE` 和 `APP_REMOTE_CONTROL_STATE_LOST` 都会在任务层回到安全输出，避免上层继续消费旧摇杆值。
4. `appRemoteControlGetOutput()` 返回的是快照拷贝，避免外部直接拿内部指针带来的撕裂读风险。
5. 这层现在适合放在 `Application`，上层业务只需要关心状态和命令，不必再直接盯原始摇杆量。
2026-03-28 关于遥控任务等待策略

1. 为了让 `LOST` 状态真的能被检测到，遥控任务不再永久 `portMAX_DELAY` 阻塞，而是改成了短超时等待。
2. 这样在正常有帧时仍然是事件驱动的，在链路断开时会周期性醒来并把输出切到安全态。
3. 这比“永远等下一帧”更适合 App 层状态机，因为业务层需要区分“正常控制”“failsafe”和“失联”。
2026-03-28 关于遥控命令层的简单实现

1. `app_remote_control` 继续只负责把 SBUS / 摇杆值整理成稳定快照。
2. 新增了一个很轻的业务命令视图：失联/失控时自动回到安全态，CONTROL 状态下再把摇杆翻译成 `armed / mode / throttle / yaw_rate / pitch_rate / roll_rate`。
3. 这层逻辑放在 App 层而不是 Module 层，后续业务代码可以直接消费命令，不需要自己再解释开关和摇杆。
2026-03-28 关于 BMI088 设备层是否注册回调

2. 设备层主要负责拉 CS、读写寄存器、读取原始加速度/角速度数据，不直接假定自己独占 SPI。
3. 如果以后确认 IMU 独占总线，或者要切到 DMA/IT 异步读数，再把回调和事务调度上移到更合适的上层。

2026-03-28 关于 BMI088 设备层接口设计

1. `device_bmi088` 的定位是设备驱动，不做姿态融合、滤波、控制业务，也不默认独占 SPI。
2. 核心职责是：SPI 读写寄存器、两路 CS 控制、原始数据读取、设备自检/配置。
3. 最小可落地接口建议先做：`Init / SoftReset / ReadChipId / ConfigAccel / ConfigGyro / ReadRaw`。
4. 内部再拆出：`ReadReg / WriteReg / SelectAccel / SelectGyro / DeselectAll` 这类私有辅助函数。
5. 先不注册 SPI 回调，等后面确认独占总线或切 DMA/IT 异步读数，再把回调和事务调度上移到 `app_imu` 或总线层。
2026-03-29 关于为什么 SBUS/RCMapper 不依赖 Bsp，而 BMI088 要依赖 Bsp

1. 根本原因不是它们都在 `Module` 层，而是职责边界切得不一样。
2. `sbus` 和 `rc_mapper` 现在被切成“纯数据变换模块”：前者只把字节流解析成 SBUS 帧，后者只把 SBUS 帧映射成摇杆/开关量，因此输入已经是内存里的数据，不必碰 UART/GPIO。
3. `BMI088` 如果被定义成“设备驱动”，它就不是纯解析器，而是必须主动发起寄存器读写、拉 CS、读取原始数据，所以天然要依赖 `bsp_spi` / `bsp_gpio` 这类总线与引脚适配层。
4. 也就是说，区别不在 `Module`，而在“这个模块是纯算法/协议解析”还是“设备驱动”。
5. 如果以后你把 `SBUS` 也做成一个拥有 UART 接收与缓冲的完整驱动，它同样可以依赖 `bsp_uart`；反过来，如果把 BMI088 再拆成纯寄存器编解码子层，那那一小层也可以不依赖 Bsp。
2026-03-29 关于是否把 SBUS 做成完整接收驱动

1. 把 SBUS 做成依赖 `bsp_uart` 的完整接收驱动不是不好，但更适合放成“driver 层”而不是直接把纯 `protocol_sbus` 做重。
2. 更稳的做法是保留 `protocol_sbus` 作为纯解析器，再在上面加一层 `device_sbus` 或 `app_remote_control` 去负责 UART、回调、缓冲和超时。
3. 这样既保留了解析器的复用性和可测试性，也给工程里一个真正能直接用的 SBUS 接收模块。
4. 对当前工程来说，不建议现在就把现有 `protocol_sbus` 和 `rc_mapper` 全部并回去；分层已经比较清楚，继续顺着这个方向更稳。

2026-03-29 关于 BMI088 从启动到读出一组 IMU 数据的链路

1. 上电后 gyro 由 `PS` 引脚决定接口并默认进 normal mode；accel 则默认是 I2C + suspend。
2. SPI 模式下，accel 必须先通过一次 dummy read 触发 `CSB1` 上升沿，切到 SPI；首次读到的值无效。
3. 然后软件需要完成 accel/gyro 的配置：确认 chip id、使能 accel、设置 range/ODR/bandwidth，必要时再开 data ready 中断。
4. 真正读一组 IMU 数据时，本质上是两次独立总线事务：一次读 accel XYZ，一次读 gyro XYZ，因为 BMI088 内部就是两个独立寄存器域和两路 CS。
5. accel 的 SPI 读有 dummy byte，读 XYZ 时要从 `0x12` 开始读 7 字节并丢掉第 1 字节；gyro 则从 `0x02` 开始直接读 6 字节。

按你现在的 SPI + 两路 CS 思路讲，BMI088 从启动到读出“完整一组 IMU 数据”，本质上是两条链路拼起来的：

加速度计一条
陀螺仪一条
因为 BMI088 不是一个单寄存器块，而是 accel + gyro 两个独立寄存器域，各自有自己的 CS、寄存器表、启动状态和读数流程。
所以你说的“一组 IMU 数据”，软件上通常指：

ax, ay, az
gx, gy, gz
而不是一次总线事务就能全读出来。

步骤

先满足硬件前提。
在 VDD / VDDIO 稳定后，gyro 的接口由 PS 引脚在上电时决定；如果你板子走的是 SPI，那 PS 必须是 SPI 对应电平。两路 CS 初始都应该保持未选中。

明确上电后的默认状态。
根据 Bosch 官方手册，gyro 上电后默认就在 normal mode；accel 上电后默认是 suspend mode，而且默认先以 I2C mode 启动，不会直接进入 SPI。

先把 accelerometer 切到 SPI。
这是 BMI088 最容易漏的一步。accel 只有在检测到 CSB1 上升沿后才会切到 SPI，所以软件初始化阶段通常要先做一次 dummy SPI read，常见做法是读 ACC_CHIP_ID。
这次读出来的值是无效的，它的目的不是拿数据，而是让 accel 从 I2C 切到 SPI。

再分别读两个 chip id，确认总线和片选没问题。
常见检查是：

ACC_CHIP_ID 应该是 0x1E
GYRO_CHIP_ID 应该是 0x0F
如果这里读不对，后面的配置和读数都别继续。

让 accelerometer 进入 normal mode。
手册给的上电切 normal 流程是：

上电后先等 1 ms
写 ACC_PWR_CTRL = 0x04
再等 450 us
这一步之后 accel 才算真正开始工作。
另外手册还要求，写寄存器之间要满足内部同步时间：normal mode 下至少 2 us，suspend mode 下至少 1000 us。

配置 accelerometer 的测量参数。
这一步通常至少要写：

ACC_CONF：ODR / bandwidth
ACC_RANGE：量程
你要先决定自己想要什么采样率、带宽和量程，不然读出来只是默认配置下的数据。

配置 gyroscope 的测量参数。
gyro 虽然上电默认是 normal，但你通常还是要显式配置：

GYRO_RANGE
GYRO_BANDWIDTH
如果后面想用 data ready 中断，还要配 GYRO_INT_CTRL 和中断映射寄存器。

等待第一批“有效数据”真的出来。
这里 accel 和 gyro 时间不一样：

accel 从 suspend 到第一笔有效样本，手册给的是 1 ms
gyro 从 power-off 到稳定到最终值附近，手册给的是 30 ms
所以工程上通常是：

配完寄存器后再等一小段稳定时间
或者轮询 data-ready 位，确认新数据真的到了再读
判断“什么时候该读”。
你有两种常见策略：

轮询状态位
用 data-ready 中断
轮询时常看：

accel 的 ACC_STATUS 或 ACC_INT_STAT_1
gyro 的 GYRO_INT_STAT_1
如果只是先把驱动跑通，轮询最简单。

读 accelerometer 的 XYZ。
accel 的数据寄存器是：

0x12 ACC_X_LSB
0x13 ACC_X_MSB
0x14 ACC_Y_LSB
0x15 ACC_Y_MSB
0x16 ACC_Z_LSB
0x17 ACC_Z_MSB
但 SPI 下 accel 有个非常关键的特性：读操作前面会先吐一个 dummy byte。
所以如果你要从 0x12 开始读 XYZ，通常要做的是：

选中 accel_cs
发起从 0x12 开始的 burst read
总共收 7 个字节
丢掉第 1 个 dummy byte
后面 6 个字节才是 X/Y/Z 原始数据
读 gyroscope 的 XYZ。
gyro 的数据寄存器是：

0x02 RATE_X_LSB
0x03 RATE_X_MSB
0x04 RATE_Y_LSB
0x05 RATE_Y_MSB
0x06 RATE_Z_LSB
0x07 RATE_Z_MSB
gyro 这边没有 accel 那个 dummy byte 的麻烦，所以通常就是：

选中 gyro_cs
从 0x02 连读 6 字节
按 little-endian 拼成三个 int16_t
把原始字节拼成 int16_t。
accel 和 gyro 都是低字节在前、高字节在后，所以一般是：

x = (msb << 8) | lsb
y = (msb << 8) | lsb
z = (msb << 8) | lsb
然后把它们解释成有符号 16 位数。

按当前量程把 raw 值换算成物理量。
这一步通常不一定放在 device_bmi088，但流程上它就在读数之后。

accel: raw -> g 或 m/s^2
gyro: raw -> deg/s 或 rad/s
换算系数取决于你刚才写进去的 ACC_RANGE 和 GYRO_RANGE。

最后把这两次读出来的结果拼成“一组 IMU 数据”。
也就是类似：

imu_raw.ax
imu_raw.ay
imu_raw.az
imu_raw.gx
imu_raw.gy
imu_raw.gz
这就是上层 app_imu 或融合算法真正要吃的那份原始输入。

一个很实用的软件视角
如果你之后写 device_bmi088，从接口角度看，启动到读数大概就是这条链：

deviceBMI088Init()
-> accel dummy read, switch to SPI
-> read accel/gyro chip id
-> write ACC_PWR_CTRL
-> config ACC_CONF / ACC_RANGE
-> config GYRO_RANGE / GYRO_BANDWIDTH
-> optional: config data-ready interrupt

deviceBMI088ReadRaw()
-> optional: poll drdy
-> read accel xyz
-> read gyro xyz
-> pack to one raw imu sample
几个最容易踩坑的点

accel 上电后不是直接 SPI，第一次 dummy read 只是切接口，不是拿有效数据。
accel 的 SPI 读有 dummy byte，gyro 没有，这两个驱动逻辑不能写成完全一样。
“完整一组 IMU 数据”不是一次 SPI 事务，而是两次独立事务。
gyro 虽然上电默认 normal，但不代表可以跳过 range/bandwidth 配置。
如果你想让 ax/ay/az 和 gx/gy/gz 时间更对齐，后面要再考虑 data-ready 中断、时间戳或同步策略；裸读只能做到“很接近”，不是严格原子同步。

2026-03-29 关于怎么从 BMI088 寄存器表判断 MSB/LSB

1. 只看控制寄存器那张表是看不出 XYZ 数据的 MSB/LSB 的，因为那张表只描述单字节控制位。
2. 判断多字节数据高低位，要去看数据寄存器名本身，例如 `ACC_X_LSB / ACC_X_MSB`、`RATE_X_LSB / RATE_X_MSB`。
3. 表头里的 `bit7 ... bit0` 只是说明“当前这个寄存器字节内部”的位意义，不是在说明跨寄存器的高低字节顺序。
2026-03-29 关于 BMI088 datasheet 里哪里明确写了 MSB/LSB

1. 手册明确写了“传感器数据由 MSB upper part 和 LSB lower part 组成”，见 4.2 Sensor Data。
2. 手册也明确给了拼接公式：`Accel_X_int16 = ACC_X_MSB * 256 + ACC_X_LSB`，gyro 同理。
3. 但对于 SPI `First Bit`，手册没有直接写出“must be MSB first”这句原话；更准确地说，是 SPI 命令位定义按标准字节位序给出，因此工程上配置成 MSB first。
4. 所以“数据寄存器高低字节顺序”是手册明确说明的；“CubeMX First Bit 选 MSB”更多是根据 SPI 帧位定义作出的实现判断。
2026-03-29 关于 BMI088 模块初始化开头的结构判断

1. `deviceBMI088Init()` 里直接用 `bspBoardGetSPIInstance(BSP_SPI_IMU)` 和固定 GPIO ID 去取资源，短期能用，但会让设备模块直接绑定当前板级命名，不利于复用。
2. 如果希望 `device_bmi088` 保持“设备驱动”而不是“板级专用驱动”，更稳的是由外部传入 `spi_instance / accel_cs / gyro_cs`，设备层只保存这些句柄。
3. `Init` 最好返回状态而不是 `void`，否则现在这种中途 `return` 会留下半初始化实例，上层不容易判断是否成功。
4. 现在这种写法适合先快速 bring-up；如果准备长期保留，建议尽早改成 `config + status return` 形式。
2026-03-29 关于 BMI088 实例结构体要不要隐藏

1. 现在这版 `config + instance` 方向是对的，`config` 负责把外部资源传进来，设备层不再直接写死 `bspBoardGetXXX()`。
2. `deviceBMI088Instance_t` 是否隐藏，关键看后面是否还要在里面加入内部状态、临时缓冲、错误标志、配置缓存等私有字段。
3. 如果只是保存 `spi / accel_cs / gyro_cs` 三个句柄，不隐藏也可以；但如果准备把它作为长期设备驱动保留，更推荐隐藏实例结构体，只在 `.c` 里定义。
4. 隐藏实例结构体的好处是能保护内部状态不被外部误改，也方便后面扩字段而不破坏接口；代价是实例内存不能再由外部直接静态创建，需要改成模块内部创建或额外提供静态内存方案。
2026-03-29 关于 BMI088 隐藏实例结构体后如何创建实例

1. 不建议因为要隐藏 `deviceBMI088Instance_t`，就直接在 `bmi088.c` 里做静态单例；这会把设备驱动迅速做成板级专用单实例驱动。
2. 如果想隐藏结构体，更稳的办法是模块内部维护一个静态实例池或静态内存池，然后提供 `Create/Init` 接口返回句柄。
3. 如果当前目标是先把驱动写通、又不想引入实例池复杂度，那就先不隐藏 `instance`，继续让外部静态创建更实际。
4. 对当前工程，推荐顺序是：先公开 `config + instance` 跑通 BMI088；等内部状态变复杂后，再决定是否收口成 opaque handle + static pool。
2026-03-29 关于设备初始化时是否需要全局关中断

1. 不建议为了防止初始化被打断，就在整个 BMI088 初始化过程中全局关中断；因为初始化里往往需要延时，而 `HAL_Delay()` 依赖 SysTick，中断关掉后可能直接卡死。
2. 更稳的做法不是“全局关中断”，而是“初始化完成前不要启用相关外设中断/回调/任务入口”。
3. 对 BMI088 这类同步驱动，初始化期间只做阻塞 SPI 访问和寄存器配置即可；等芯片配置完成、实例状态稳定后，再注册或使能 data ready EXTI 等中断源。
4. 如果担心实例半初始化被别的上下文看到，可以用 `initialized_` 标志、局部临时变量、或最后一步再发布句柄，而不是长时间屏蔽全局中断。
2026-03-29 关于别人用全局关中断包住初始化的做法

1. 这种写法不是绝对错误，但它依赖很强的前提：初始化阶段不使用依赖 SysTick/中断的延时，不启动需要中断驱动的外设流程，并且只在很短的 bring-up 窗口内使用。
2. 代码注释里已经把前提写出来了：不要在初始化过程中使用中断和常规延时函数；若必须，只允许用不依赖中断的 `DWT_Delay()`。
3. 这更像一种“裸机式、强约束的启动策略”，适合项目整体都按这个约束设计；不适合作为通用规则直接套到所有设备初始化上。
4. 对当前 BMI088 这种会涉及寄存器配置、可能需要等待稳定时间的设备，更稳的办法仍然是：初始化期间不要启用相关中断/任务，而不是长时间全局关中断。
2026-03-29 关于通用设备驱动里是否直接使用 HAL_Delay

1. 更合理的结论是：即使不中断屏蔽，通用 `device` 驱动里也不建议直接调用 `HAL_Delay()`。
2. 原因不是单纯怕卡住，而是 `HAL_Delay()` 会把设备驱动绑定到 STM32 HAL、SysTick 和当前系统时基实现，降低驱动的独立性与可复用性。
3. 如果设备驱动确实需要等待，更稳的做法是通过 `config` 传入 `delay_us / delay_ms` 回调，或由更上层在初始化步骤之间负责等待。
4. 对当前 BMI088，推荐优先用“驱动依赖抽象 delay 回调”的方式，而不是在 `deviceBMI088Init()` 里直接写死 `HAL_Delay()`。
2026-03-29 关于当前 DWT 延时实现的关键问题

1. `DWT_BASE |= ...` 是错的，应该操作的是 `DWT->CTRL`，不是基地址常量。
2. `bspDWTDelayUs()` 里的周期换算公式写反了，正确思路应是“目标周期数 = SystemCoreClock * time_us / 1000000”。
3. 延时循环里必须反复读取 `DWT->CYCCNT`，不能只让局部变量自增，否则根本不是按硬件周期在等。
4. `DelayUs()` 不应该每次都 reset 全局 `CYCCNT`，否则会破坏 DWT 计数器作为全局时间基/测量基准的用途。
5. 如果继续走这条路，建议把 `bspDWT` 做成板级无中断延时能力：`Init / GetCount / DelayUs / DelayMs`，内部用 `DWT->CTRL` 和 `CYCCNT` 正常实现。
2026-03-29 关于 DWT 延时函数能否兼作全局计数器

1. 现在这版 `bspDWTDelayUs()` 不再 reset `CYCCNT`，因此不会破坏 DWT 作为全局自由运行计数器的用途。
2. 但 `CYCCNT` 只有 32 位，不是任意时长都能直接表示；在 84MHz 下单次可安全表示的大约是 51.1 秒。
3. 用无符号减法比较差值可以正确跨一次回绕，因此短到中等时长的延时/测量没有问题。
4. 如果要做真正长时间的全局时间基，仍然更适合再扩成软件 64 位计数，或改用专门的硬件定时器。
2026-03-29 关于 DWT 计数转微秒函数的实现细节

1. 计算 `current_cnt` 与 `start_cnt` 的差值时，其实直接用无符号减法 `current_cnt - start_cnt` 就够了，天然支持单次回绕。
2. 手写分支版如果走回绕路径，要小心少算 1 个周期；更简单也更稳的是不要自己拼 `UINT32_MAX - start + current`。
3. 把周期数转成微秒时，要先把 `elapsed_cnt` 转成 `uint64_t` 再乘 `1000000U`，否则会先在 32 位里溢出。
4. 这类函数仍然只适用于不超过一次 32 位回绕的时间范围；84MHz 下大约是 51 秒。
2026-03-29 关于 BMI088 初始化的推荐顺序

1. `deviceBMI088Init()` 更适合按 bring-up 顺序写：资源绑定 -> CS 释放 -> accel dummy read 切 SPI -> 读 chip id -> 软复位/等待 -> 配置 accel/gyro -> 标记初始化成功。
2. BMI088 的关键特殊点是：gyro 上电后按 `PS` 引脚接口启动，而 accel 默认先在 I2C 模式，SPI 下必须先通过一次 dummy read 让 `CSB1` 上升沿把 accel 切到 SPI。
3. accel 进入 normal mode 需要写 `ACC_PWR_CTRL = 0x04`，并满足手册要求的等待时间；gyro 则主要配置 range / bandwidth。
4. 真正的 `ReadRaw()` 不应塞进 `Init()`，初始化只负责把设备带到“后续可稳定读数”的状态。
5. 如果当前先做最小版本，建议初始化只验证 `ACC_CHIP_ID / GYRO_CHIP_ID`、完成基本配置，并把 data ready / 中断配置留到后面。
2026-03-29 关于先实现 BMI088 初始化的最小流程

1. 推荐先做同步、阻塞式 bring-up，不上中断、不上 DMA、不做自检，只做“资源绑定 -> 切 accel 到 SPI -> 读 chip id -> 配置 accel/gyro -> 标记初始化完成”。
2. 初始化顺序可以先收敛成：保存 `spi/cs` 句柄，拉高两路 CS，dummy read 一次 `ACC_CHIP_ID` 切 accel 到 SPI，再正式读取 `ACC_CHIP_ID=0x1E` 和 `GYRO_CHIP_ID=0x0F`。
3. 然后配置 accel：必要时写 `ACC_PWR_CTRL=0x04` 进入 normal mode，并满足 `1ms + 450us` 的启动等待；再写 `ACC_CONF / ACC_RANGE`。
4. 再配置 gyro：写 `GYRO_RANGE / GYRO_BANDWIDTH`，若做 soft reset 则要额外等待 `30ms`。
5. 最后可选地读一次 `ACC_STATUS / GYRO_INT_STAT_1` 或直接试读一帧 raw 数据，确认初始化链路打通。
2026-03-29 关于当前 BMI088 源码骨架能否继续往下写

1. 当前 `hidden struct + static instance pool + config 传入资源句柄` 的方向可以继续往下写，整体框架已经比最早那版更合理。
2. 但最好先补一条公开前置声明 `typedef struct device_bmi088 deviceBMI088Instance_t;`，否则头文件里外部没法正确使用这个 opaque type。
3. `deviceBMI088InstanceInit()` 和 `deviceBMI088Init()` 的职责已经开始分成“创建实例”和“真正初始化设备”，这个方向是对的，但命名最好更明确，比如 `Create`/`Init`。
4. 在继续写寄存器访问前，最好先把底层私有 helper 形状定下来：`SelectAccel / SelectGyro / DeselectAll / ReadRegs / WriteRegs`，否则后面 `WriteSingleRegister` 很容易越写越散。
2026-03-29 关于 BMI088 的建议函数清单

1. 在当前 `target enum + static pool + opaque instance` 结构下，公共接口建议先保留 `Create / Init / ReadChipId / ReadRaw` 这几项。
2. 私有辅助函数建议围绕 `SelectTarget / DeselectAll / ReadRegs / WriteReg / ConfigAccel / ConfigGyro / AccelSwitchToSPI` 展开。
3. 第一版先把 bring-up 链路打通，不急着做中断、自检、温度、校准和 DMA/IT。
2026-03-29 关于 BMI088 寄存器读写 helper 的写法

1. `deviceBMI088ReadRegisters()` 这类 helper 最好自己完整管理一次事务：选中目标 CS -> SPI 收发 -> 释放 CS，不要依赖外部预先选中。
2. 当前 `bsp_spi` 的同步 `TransmitReceive` 已经够 bring-up 使用，不必为了 BMI088 先改 `bsp_spi`。
3. 读寄存器时可以在 helper 内部构造小的临时 `tx/rx` 缓冲区；BMI088 第一版单次事务很短，通常 `data_length + 1` 就够。
4. 不建议通过 `bspGPIOReadLogic()` 判断“CS 是否已选中”；同步驱动里应由 helper 自己控制和记录事务边界。
5. 另外要注意 SPI 命令字节不是直接把寄存器地址原样发出去，而是要按 BMI088 的 SPI 帧格式拼出读/写命令。

2026-03-29 BMI088初始化补充：
1. 当前 `deviceBMI088Init()` 按最小 bring-up 顺序实现为：释放 CS -> 等待总线空闲 10us -> 等待上电稳定 30ms -> accel dummy read 切 SPI -> 校验 accel/gyro chip id -> 配置 accel -> 配置 gyro。
2. accel 读取链路和 gyro 不同，SPI 读 accel 时在寄存器地址后还有 1 个 dummy byte，所以读寄存器临时缓冲至少要预留 `地址 + dummy + 数据`，不能只按 `地址 + 数据` 分配。
3. 这版初始化先不做 SPI 回调、不假定独占 SPI，也没有额外加 soft reset，先把官方上电访问和基础寄存器配置走通。
4. 当前配置值是：`ACC_PWR_CONF=ACT`、`ACC_PWR_CTRL=ON`、`ACC_CONF=0xA8`、`ACC_RANGE=6G`、`GYRO_RANGE=2000dps`、`GYRO_BANDWIDTH=2000Hz/532Hz`。
2026-03-29 BMI088 UpdateData补充：
1. `deviceBMI088UpdateData()` 当前实现为两次阻塞寄存器读取：一次从 `ACC_X_LSB_ADDR` 连续读 6 字节，一次从 `GYRO_RATE_X_LSB_ADDR` 连续读 6 字节，然后按 `MSB << 8 | LSB` 拼成三轴 16 位原始值。

2026-03-29 BMI088单位与延时修正：
1. `deviceBMI088Data_t` 里的 `accel_raw_` / `gyro_raw_` 更合理的类型是 `int16_t`，因为 BMI088 三轴原始数据本来就是有符号 16 位值。
2. 当前 `accel_` 已按 `m/s^2` 输出，`gyro_` 已按 `rad/s` 输出；其中 gyro 的 `±2000dps` 分辨率按 datasheet 表给出的 `16.384 LSB/(deg/s)` 计算，再乘 `pi/180` 转成 `rad/s`。
3. accel 这里采用的是寄存器公式等效出来的 `±6g` 理想映射：`32768 / 6 = 5461.333... LSB/g`，再乘标准重力加速度 `9.80665m/s^2` 转成 SI 单位；这和 datasheet 典型值 `5460 LSB/g` 只有很小差别。
4. `DEVICE_BMI088_POWER_ON_DELAY_US = 30000U` 的依据是 gyro 上电启动时间取 30ms 的保守值；当前初始化是整颗 BMI088 一起 bring-up，所以直接按较慢的 gyro 约束等一次，accel 的 1ms 要求也自然被覆盖了。
5. 原先 `DEVICE_BMI088_POWER_MODE_SWITCH_DELAY_US = 5000U` 没有很强的官方依据，现已改成 `450U`；这个值对应 accel 从 suspend/配置上电阶段常见的等待量级，至少比 5ms 更贴近 datasheet。
2026-03-29 BMI088 ACC_PWR_CONF 与 ACC_PWR_CTRL 区别：
1. `ACC_PWR_CONF = 0x00` 不是“真正进入 normal mode”的决定性动作，它更像是把 accel 的 power-save/filter 配置切到 active。
2. 真正控制 accel 是否开始正常工作的，是 `ACC_PWR_CTRL`；datasheet 明确写的是“要进入 normal mode，向 ACC_PWR_CTRL 写 0x04”。
3. 所以当前初始化里更准确的理解是：先把 `ACC_PWR_CONF` 设到 active，再满足 suspend 状态下写间隔要求，然后由 `ACC_PWR_CTRL = ON` 真正把 accel 拉进 normal mode。

2026-03-29 关于 BMI088 下一步要不要上 EXTI + DMA：
1. 对 BMI088 这类“小包高频采样”设备，`EXTI data-ready + 阻塞SPI读取` 往往先比 `EXTI + DMA` 更值得做，因为一帧只读十几字节，DMA 的收益不一定能覆盖回调、状态机和总线仲裁复杂度。

2026-03-29 关于 BMI088 DMA/EXTI、INS_TASK 和磁力计同步：
1. accel 和 gyro 的“同步”不建议靠 EventGroup 做，因为它们不是两个独立生产者任务，而是同一个 IMU 采样事务里的两个连续 SPI 子事务；更合适的是由同一个任务在同一个 data-ready 触发下顺序完成 accel DMA 和 gyro DMA，并给整组数据打同一个时间戳。
2. 在 FreeRTOS 下，这里比 EventGroup 更适合用 direct task notification：EXTI ISR 通知 `ins_task/app_imu`，任务启动 accel DMA；SPI 完成回调再通知同一个任务继续启动 gyro DMA。
3. 如果磁力计后面也要进 INS，不建议让高频 IMU 任务每次都阻塞等待“磁力计也来一帧”；更合理的是磁力计独立更新 `latest mag sample + timestamp`，INS 在 IMU 周期里取“最近且足够新鲜”的磁力计样本融合。

5. 当前 `bsp_spi` 的完成回调模型是“一个 SPI 实例只有一个 owner/callback”，所以如果未来多个设备要共享同一个 SPI 并且都走异步回调，现有 BSP 不够，需要在上层加一个 bus manager / transaction scheduler。

2026-03-29 关于 bsp_spi 要不要像 bsp_can 一样做 route：

- 2026-03-30 关于初始化阶段是否全局关中断：不建议把“整个 RobotInit/AppInit”都包在 `__disable_irq()` 里当默认做法。更稳的是按资源逐步初始化，并把“注册 owner/callback -> 记录 task handle/buffer/context -> 最后再开启对应外设接收/中断源”作为基本顺序。全局关中断只适合极短的临界区；如果范围过大，容易影响 HAL 超时/时间基准、掩盖初始化顺序问题，也不利于以后扩展。

- 2026-03-30 `app_imu` 数据全 0 排查：`appIMUUpdate()` 里原先 `memcpy(..., 3U)` 只复制了 3 个字节，而不是 3 个 `float`，这会直接把 `appIMUData_t` 写坏；应改成 `sizeof(array)`。同时 `appIMUInit()` 里虽然注册了 EXTI callback，但原先漏掉了 `deviceBMI088ConfigGyroDataReadyIT()`，导致 BMI088 的 gyro data-ready 中断配置链路不完整；顺序应为“注册 callback/owner -> 再使能传感器中断”。
- 2026-04-01 `remote_control` 结构复查：把任务循环从 `freertos.c` 收进 `app_remote_control.c` 这个方向是对的，`freertos.c` 更接近只做线程创建入口了。当前需要注意两点：1) `appRemoteControlUpdate()` 在“持续收到字节但始终拼不出合法 SBUS 帧”时不会把状态切到 `LOST/FAILSAFE`，可能保留旧控制量；2) `appRemoteControlInit()` 仍是 `void` 且忽略 `bspUARTRxStart()` 结果，初始化失败时任务会静默空转。
- 2026-04-01 `remote_control` 修正落地：1) `appRemoteControlInit()` 改为返回 `bool`，`TaskEntry` 在初始化失败时按固定周期重试，不再静默空转；2) 丢控判定改成“距离上一次有效 SBUS 帧超过阈值则输出 `LOST`”，这样既不会因为分包/半帧立刻误判，也不会在持续坏流时一直保留旧控制量；3) `app_remote_control.h` 只保留 `GetState/GetCommand/TaskEntry` 这类 App 级接口，不再直接暴露 `moduleRCMapper_t`，更适合后续由 commander / mode manager 消费。
- 2026-04-01 `remote_control` 小清理：在丢控逻辑改成“基于上次有效帧超时”后，`received_any_byte` 已不再参与判断，属于残余变量，应删除，避免误导后续阅读。
- 2026-04-01 关于 `remote_control` 的有效帧时间戳：`last_valid_frame_tick_` 应该在“成功解析出一帧合法 SBUS 帧”时更新，而不是在“收到任意字节”时更新，这个方向是对的；否则坏流会一直续命。但更稳的实现应把“阻塞等待多久”和“多久判定 LOST”分成两个独立参数，不要直接复用同一个 `timeout_tick`。另外，时间戳最好在 `FRAME_OK` 分支当场记录，而不是在整个消费循环结束后再统一记录，语义更清楚。
